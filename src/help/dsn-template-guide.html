What DSN Rules Are

DSN means Detection Specification Notation. A DSN ruleset is safe JSON that tells Sanitas how to detect extra values before the normal Clean workflow replaces them with placeholders.

Use DSN when the built-in detectors are not enough. For example, you can hide exact project names, values after labels, tokens with known prefixes, structured internal IDs, or short codes that are only sensitive near a trusted label.

Why DSN Does Not Allow Raw Regex

Raw regex import is not allowed because complex patterns can be unsafe, confusing, slow, or too broad. DSN gives users controlled rule types instead: literal text, labels, prefixes, fixed segments, and trusted context labels. There is no JavaScript, no executable logic, no remote URL loading, and no raw regex field.

Quick Start

  1. Choose the right rule type. Use the decision table below to match your requirement to one of the five supported rule types.
  2. Insert a template. Open the Sanitas floating window, choose the DSN Rules tab, and click the matching template button.
  3. Edit only the safe fields. Change labels, prefixes, terms, segment lengths, placeholder type, and priority. Do not add pattern, regex, script, URL, or code fields.
  4. Validate Rules. Click Validate Rules. If validation fails, fix the listed path and message.
  5. Apply to Current Session. Validation alone does not activate the rules. Apply stores the validated rules for the current browser session only.
  6. Test in the Text tab. Switch to Text, choose Clean, paste dummy input, and confirm the output contains placeholders.
  7. Clear Session Rules when finished. Clear active DSN rules when testing is done. The main Clear button also removes DSN rules, mappings, custom terms, and workspace text.

Complete Field Reference

Every rule has common fields. Each rule type also has its own type-specific fields.

Top-Level Ruleset Fields

Field Required What it does Allowed values Example value Common mistakes
schema_version Yes Declares the DSN schema version. "1.0" "1.0" Using a number or unsupported version.
dsn_type Yes Identifies the JSON as a Sanitas safe ruleset. "sanitas_safe_rules" "sanitas_safe_rules" Omitting it or changing the name.
ruleset_id Yes Gives the ruleset a safe identifier. Letters, numbers, underscore, dot, dash; starts with a letter or number. "safe-demo-rules" Spaces or symbols such as #.
name No Human-readable ruleset name. Short text. "Safe Demo Rules" Putting private values in the name.
version Yes Ruleset version label. Non-empty text up to 40 characters. "1.0.0" Leaving it blank.
description No Short explanation for the ruleset. Text up to the supported limit. "Dummy DSN examples." Including private values in the description.
rules Yes Array of rule objects. An array; up to the supported rule count. [{ ... }] Using an object instead of an array.

Common Rule Fields

Field Required What the field does Allowed values Example value Common mistakes
id Yes Unique rule identifier used in validation and debugging summaries. Starts with a letter; letters, numbers, and underscores only. "PROJECT_NAMES" Spaces, dashes, or starting with a number.
type Yes Selects the matching strategy. literal_terms, labeled_value, prefix_token, structured_id, context_value "prefix_token" Using unsupported names such as regex.
enabled Yes Turns the rule on or off. Boolean true or false. true Using "true" as a string.

Full Tutorial For Each Rule Template

literal_terms

Use case: Hide exact project names, customer names, product codenames, or organization names you already know.

{
  "id": "PROJECT_NAMES",
  "type": "literal_terms",
  "enabled": true,
  "placeholder_type": "CUSTOM_TERM",
  "priority": 500,
  "terms": ["Project Titan", "Acme Corp"],
  "case_sensitive": false,
  "match_word_boundary": true
}

Input text before sanitization

We discussed Project Titan with Acme Corp.

Output text after sanitization

We discussed [[CUSTOM_TERM_0001]] with [[CUSTOM_TERM_0002]].
  • terms: array of exact words or phrases. Minimum length is 3 characters. Empty values are not allowed.
  • case_sensitive: false matches Project Titan and PROJECT TITAN; true requires exact case.
  • match_word_boundary: true prevents matching inside a larger word. Use false only when the term may be embedded.
  • placeholder_type: controls tokens such as [[CUSTOM_TERM_0001]].

When to use it: use it for known exact text. When not to use it: do not use it for values that vary by prefix, label, or structured numeric segments. Common mistakes: misspelling the term, making the term too short, or setting match_word_boundary to true when the text is attached to punctuation-like characters you did not expect.

labeled_value

Use case: Detects values after trusted labels. It replaces only the value, not the label. Good labels include DB_PASS=, API_SECRET:, TOKEN=, and CLIENT_ID=.

{
  "id": "DB_PASSWORD",
  "type": "labeled_value",
  "enabled": true,
  "placeholder_type": "DATABASE_SECRET",
  "priority": 900,
  "labels": ["DB_PASS", "DATABASE_PASSWORD"],
  "separators": ["=", ":"],
  "value_mode": "until_whitespace",
  "case_sensitive": false
}

Input text before sanitization

DB_PASS=SafeDummyValue123

Output text after sanitization

DB_PASS=[[DATABASE_SECRET_0001]]
  • labels: trusted words that appear before the value. Labels must be strings.
  • separators: strings that connect the label to the value, such as = or :.
  • value_mode: until_whitespace, until_line_end, quoted_value, or token.
  • case_sensitive: false lets db_pass match DB_PASS.

When to use it: use it when the label is stable and trusted. When not to use it: do not use it for unlabeled values or labels that appear in normal prose. Common mistakes: forgetting the separator, using the wrong value mode, or expecting the label itself to be replaced.

prefix_token

Use case: Detects tokens that start with known prefixes. Good for API keys, test keys, and internal tokens.

{
  "id": "TEST_API_KEY",
  "type": "prefix_token",
  "enabled": true,
  "placeholder_type": "API_KEY",
  "priority": 950,
  "prefixes": ["sk_test_", "pk_test_"],
  "min_length": 16,
  "max_length": 120,
  "allowed_chars": "alpha_num_underscore_dash",
  "case_sensitive": true
}

Input text before sanitization

The key is sk_test_ABCDEF1234567890.

Output text after sanitization

The key is [[API_KEY_0001]].
  • prefixes: known token starts such as sk_test_.
  • min_length and max_length: total token length, including the prefix.
  • allowed_chars: supported values are alpha_num, alpha_num_dash, alpha_num_underscore, alpha_num_underscore_dash, base64url, hex, and token_safe.
  • case_sensitive: usually true for keys because prefixes often have exact case.

When to use it: use it for tokens with known prefixes. When not to use it: do not use it for values without predictable prefixes. Common mistakes: setting min_length higher than the real token length, choosing hex for tokens that contain underscores, or forgetting that punctuation after the token is not included.

structured_id

Use case: Detects IDs with predictable format. Good for internal document IDs, ticket IDs, asset IDs, and invoice IDs.

{
  "id": "INTERNAL_DOCUMENT_ID",
  "type": "structured_id",
  "enabled": true,
  "placeholder_type": "DOCUMENT_ID",
  "priority": 700,
  "prefixes": ["DOC-INT-"],
  "segments": [
    {
      "kind": "digits",
      "min": 4,
      "max": 12
    }
  ],
  "case_sensitive": true
}

Input text before sanitization

Please review DOC-INT-445566.

Output text after sanitization

Please review [[DOCUMENT_ID_0001]].
  • prefixes: one or more fixed starts, such as DOC-INT-.
  • segments: ordered pieces after the prefix. Supported segment kinds are digits, alpha, alnum, hex, and literal.
  • For digits, alpha, alnum, and hex, use min and max.
  • For a literal segment, use value. Do not add min or max to a literal segment.

When to use it: use it for fixed formats. When not to use it: do not use it for values that are free-form or label-based. Common mistakes: forgetting a literal dash segment when the format has a second dash, or using an unsupported segment kind.

context_value

Use case: Detects weak values only when a trusted context label appears. Good for CVV, PIN, OTP, short codes, or expiration-like values. This is safer than detecting all 3-digit numbers because it only matches near configured labels.

{
  "id": "CARD_SECURITY_CODE",
  "type": "context_value",
  "enabled": true,
  "placeholder_type": "CARD_SECURITY_CODE",
  "priority": 850,
  "context_labels": ["CVV", "CVC"],
  "separators": [":", "="],
  "value_kind": "digits",
  "min_length": 3,
  "max_length": 4,
  "case_sensitive": false
}

Input text before sanitization

CVV: 123

Output text after sanitization

CVV: [[CARD_SECURITY_CODE_0001]]
  • context_labels: trusted labels such as CVV, CVC, PIN, or OTP.
  • separators: strings between label and value, such as : or =.
  • value_kind: supported values are digits, alnum, token, date_mm_yy, and date_mm_yyyy.
  • min_length and max_length: allowed length for the detected value. Date kinds have fixed formats.

When to use it: use it when a short value is sensitive only next to a trusted label. When not to use it: do not use it when every value with the same shape should be hidden everywhere. Common mistakes: wrong separator, wrong value kind, or a length range that excludes the value.

Build Rules From Requirements

This is the core workflow: describe what must be hidden, choose the safest rule type, edit one template, validate it, apply it, then test with dummy text.

Requirement: I want to hide my company internal ticket IDs like INC-2026-000123.

Best rule type: structured_id. Why: the value has a fixed prefix and controlled numeric segments.

{
  "id": "INTERNAL_TICKET_ID",
  "type": "structured_id",
  "enabled": true,
  "placeholder_type": "TICKET_ID",
  "priority": 750,
  "prefixes": ["INC-"],
  "segments": [
    { "kind": "digits", "min": 4, "max": 4 },
    { "kind": "literal", "value": "-" },
    { "kind": "digits", "min": 6, "max": 6 }
  ],
  "case_sensitive": true
}
Test input: Ticket INC-2026-000123 needs review.
Expected output: Ticket [[TICKET_ID_0001]] needs review.

Requirement: I want to hide anything after API_TOKEN=.

Best rule type: labeled_value. Why: the trusted label identifies the sensitive value.

{
  "id": "API_TOKEN_VALUE",
  "type": "labeled_value",
  "enabled": true,
  "placeholder_type": "API_TOKEN",
  "priority": 900,
  "labels": ["API_TOKEN"],
  "separators": ["="],
  "value_mode": "until_whitespace",
  "case_sensitive": false
}
Test input: API_TOKEN=SafeDummyToken123 next step.
Expected output: API_TOKEN=[[API_TOKEN_0001]] next step.

Requirement: I want to hide exact customer names or project names.

Best rule type: literal_terms. Why: the exact phrases are known before testing.

{
  "id": "KNOWN_NAMES",
  "type": "literal_terms",
  "enabled": true,
  "placeholder_type": "CUSTOM_TERM",
  "priority": 500,
  "terms": ["Project Titan", "Acme Corp"],
  "case_sensitive": false,
  "match_word_boundary": true
}
Test input: Project Titan met Acme Corp.
Expected output: [[CUSTOM_TERM_0001]] met [[CUSTOM_TERM_0002]].

Priority Explained

For DSN rules, higher numeric priority wins when two DSN matches overlap. The sanitizer resolves overlaps before replacing text. If priorities tie, the longer match is favored, then the earlier text position.

Example: an API key prefix rule should have higher priority than a generic literal term rule. If a generic term could match part of sk_test_ABCDEF1234567890, the prefix_token API key rule should use a higher priority such as 950 so the whole token becomes [[API_KEY_0001]].

Built-in detectors also participate in overlap resolution. Keep DSN rules specific and give the most sensitive, most precise rules the highest priority.

Validation Errors and Fixes

Error pattern Likely cause Fix
Missing schema_version Top-level field is absent. Add "schema_version": "1.0".
Missing rules array rules is absent or not an array. Add "rules": [ ... ].
Unsupported rule type type is not one of the five supported names. Use literal_terms, labeled_value, prefix_token, structured_id, or context_value.
Missing required field A rule lacks id, type, enabled, or placeholder_type. Add the field shown in the validation path, such as rules.0.placeholder_type.
Wrong field type, such as string instead of array A list field like terms, labels, or prefixes is written as one string. Use an array: "terms": ["Project Titan"].
min_length greater than max_length The minimum is larger than the maximum. Lower min_length or raise max_length.
Empty terms, labels, or prefixes The list is empty or only contains blank strings. Add at least one dummy-safe value while testing.
Unsupported allowed_chars The value is not in the supported list. Use alpha_num, alpha_num_dash, alpha_num_underscore, alpha_num_underscore_dash, base64url, hex, or token_safe.
Invalid JSON syntax Missing comma, extra comma, wrong quote, or unclosed bracket. Click Pretty Format JSON. If it fails, fix the JSON syntax first.

Security Rules

Keep DSN Safe

  • No raw regex.
  • No JavaScript.
  • No remote URLs.
  • No executable logic.
  • No production secrets in examples.
  • Use dummy examples only.
  • Rules are stored only for the current browser session.
  • Clear session after testing.

Troubleshooting

Problem What to check
Rule validates but does not detect text. Confirm you clicked Apply to Current Session, then test in the Text tab with Clean selected.
Case sensitivity issue. If the input case varies, set case_sensitive to false.
Wrong separator. Add the separator used in the input, such as =, :, or - .
Value too short because min_length is too high. Lower min_length or use a rule type that better fits the value.
Value too long because max_length is too low. Raise max_length within the supported range.
Wrong allowed_chars. If the token contains underscore or dash, do not use alpha_num or hex.
Forgot to click Apply to Current Session. Validate only checks JSON. Apply activates the rules for the session.
Testing in the wrong tab. Use the Text tab, choose Clean, and paste dummy input into Input.
Session was cleared. Clearing session removes DSN rules. Apply the rules again before testing.
